## 1. About Jitsu — Main Business

Jitsu (formerly known as Axlehire) is a last-mile delivery and logistics platform. The platform connects three main parties:

| Party | Role |
| --- | --- |
| **Clients (Merchants)** | Businesses that ship parcels (e.g. The Farmers Dog, Hungryroot, Whiplash). They submit shipments to Jitsu via web upload or API. |
| **Drivers** | IC, DSP, 3P drivers pick up parcels and deliver to recipients. Linehaul drivers handle first/middle mile transport. All managed via the Driver App. |
| **Recipients** | End customers who receive parcels. They can track shipments, update addresses, leave delivery instructions, and provide feedback via the Recipient web. |

### Core Delivery Lifecycle

A typical Jitsu shipment flows through these stages:

1. Client submits a shipment (single via API, or batch via CSV upload from the Client web).
2. Shipment is geocoded and validated, then the Routing engine finds shortest paths and produces Routes.
3. Routes/Assignments are made available to drivers — IC drivers book via Booking Session / Direct Booking, DSP/3P are assigned directly.
4. Driver picks up parcels at the warehouse (Inbound sort → Scan → Audit), then delivers.
5. During delivery: dimensions may be measured (Warehouse App), parcels are scanned, POD captured.
6. Recipient receives notifications and can interact via the Recipient web.
7. Driver is paid weekly through the payment system; client receives reports.

## 2. Key Terminology & Concepts

Master these definitions early — they appear in every ticket and every discussion.

| Term | Definition |
| --- | --- |
| **Shipment** | A single parcel from a client to a recipient. Each has a geocoded delivery address. |
| **Route** | The optimized path generated by the Routing engine to deliver a set of shipments — shortest path between stops. |
| **Assignment** | The set of shipments grouped together AFTER routing has computed an optimal path. An Assignment = shipments + a route + driver-ready container that can be booked or assigned. |
| **Zone** | A small geographic sub-area within a Region used for IC booking and route generation. |
| **Region** | A larger geographic market (e.g. SFO, LAX). One region contains multiple zones. ONE region has only ONE client service type. |
| **Stem miles** | Distance from the deployment facility to the driver's first stop, and from the last stop back. Minimizing stem miles is a key optimization goal. |
| **OTD** | On-Time Delivery — percentage of shipments delivered within the promised window. A primary KPI. |
| **POD** | Proof of Delivery — photo (and sometimes signature) captured by the driver at the recipient's location. |
| **SLA** | Service Level Contract — the per-client contract that defines delivery speed commitments and other terms. SLA is per-client, NOT a generic speed tier. |
| **Courier** | Jitsu-specific term for DSP or 3P drivers. NEVER used for IC drivers. |
| **Tour cost** | Base pay for a route before any bonuses. |
| **Sprinkling** | Inserting a late-arriving shipment into an existing route that has capacity, rather than creating a new route. |
| **Geofencing** | Location-based gate — driver actions like check-in or delivery only succeed when physically within an expected coordinate area. |

> **💡 Region vs Zone — Visual**
> USA → States → Regions (e.g. SFO, LAX) → Zones (sub-areas inside each region). Each region has its OWN zones — zones do not cross regions.

## 3. Warehouses, Regions & Zones

### 3.1 Warehouse Types

Jitsu uses 4 distinct types of warehouses. Knowing which type is involved in a flow tells you what test data to set up.

| Warehouse Type | Purpose |
| --- | --- |
| **Main Warehouse** | Jitsu's main fixed facility — full sortation capability (inbound, sort, audit, outbound). The hub of every region. Starting point for large routes. |
| **Mobile Hub** | Lightweight, month-to-month rented mini-warehouse located near a high-density delivery cluster. Reduces stem miles. Can be opened or closed dynamically — sometimes as late as the day before delivery. |
| **Client Warehouse** | Warehouse owned by the Client. Used as pickup location for On-Demand, Specialty, and FCTD flows (Jitsu drivers come to the Client WH to pick up). |
| **DSP Warehouse** | Warehouse owned by a DSP partner. Used as pickup point for DSP-operated routes. |

### 3.2 Region & Zone Hierarchy

Jitsu operates in approximately 29 US markets. The geographic hierarchy is:

```
USA
├── State (50 states)
│   ├── Region (e.g. SFO, LAX, NYC, DFW, ...)
│   │   ├── Zone A
│   │   ├── Zone B
│   │   └── Zone C
│   └── Region 2
│       ├── Zone X
│       └── Zone Y
└── ...
```

**Critical rules about regions and zones**

- USA is divided into states; each state can have multiple Regions.
- Each Region is divided into multiple Zones; zones do NOT cross regions.
- **ONE Region has only ONE Client Service Type.** (See Section 4.)
- IC drivers are restricted by Zone — they typically book within specific zones.
- DSP/3P drivers are NOT bound by zone boundaries — they operate on negotiated geographic areas.

> **📌 NOTE for QA setup**
> When you create a test client, you must add a `client_service` entry for each region the client serves. Since region = single service type, you can only assign a client to regions matching its type.

## 4. Client Service Types & Delivery Flows

Jitsu supports 4 client service types. When onboarding a new client, the Sales team negotiates which type the client falls into — this determines pricing and flow.

### 4.1 Two layers: `service_type` (client) vs `logistic_type` (post-routing)

There are **two distinct fields** in two different tables. Don't confuse them:

**1️⃣ `service_type` table — the client service type**

The type negotiated with the client. The main values are **string codes** (not 0–3):

| `service_type` | Meaning |
| --- | --- |
| `COMMINGLE` | Standard ship. Client delivers boxes to the Jitsu Warehouse first; Jitsu sorts and delivers. Delivery window typically 8 AM – 8 PM. Like normal e-commerce ship. |
| `ONDEMAND` | Rapid / same-day. Jitsu picks up from Client Warehouse/Store and delivers directly. Speed-priority. Higher pricing. |
| `SPECIALTY` | Special handling (cold chain, fragile, oversized). Pickup from Client. May require a specific vehicle / certified driver. |
| `FCTD` | First-Class Transit Delivery — dedicated routes for a single client (not commingled with others). Contract-priced. |

**2️⃣ `logistic_type` in the `assignments` table — set after routing**

Once routing is done, each assignment carries a numeric `logistic_type` enum (0–3):

| `logistic_type` | Code |
| --- | --- |
| `NEXT_DAYS` | 0 |
| `ON_DEMAND` | 1 |
| `SPECIALTY` | 2 |
| `FCTD` | 3 |

**Mapping `service_type` → `logistic_type`:**

| `service_type` (client) | `logistic_type` (assignments) | Code |
| --- | --- | --- |
| `COMMINGLE` | `NEXT_DAYS` | 0 |
| `ONDEMAND` | `ON_DEMAND` | 1 |
| `SPECIALTY` | `SPECIALTY` | 2 |
| `FCTD` | `FCTD` | 3 |

> **📌 COMMINGLE (service_type) = NEXT_DAYS (logistic_type, code 0)**
> The one rename to watch: the client-facing `COMMINGLE` becomes `NEXT_DAYS` (enum 0) once it lands in `assignments` after routing. The other three keep effectively the same name (`ONDEMAND` → `ON_DEMAND`). In code/DB, always reference the exact field of the table you are querying.

### 4.2 Delivery Flow per Type

**🟢 NEXT_DAYS (Commingle):** `Client → Jitsu Warehouse → Customer`

- Client ships their boxes to Jitsu's Main Warehouse.
- Jitsu sorts, routes, and delivers to the customer.
- Delivery window: 8 AM – 8 PM (standard ship).
- Always passes through Inbound Sort at the Jitsu WH.

**🟡🟠🔵 ON_DEMAND / SPECIALTY / FCTD:** `Client Warehouse / Store → Customer` (typical, direct flow) OR `Client Warehouse → Jitsu Warehouse → Customer` (when sort/cross-dock needed)

- Jitsu drivers pick up from the Client's warehouse/store.
- Typically deliver directly to the customer (faster than Commingle).
- Higher pricing — same-day or contractual SLA.
- Can also go through the Jitsu Warehouse for sort if operationally needed — not all bypass.

### 4.3 Mixing Client Types in Routes

Routes typically group shipments of the SAME client type — this is largely natural because:

- **ONE Region = ONE Client Service Type** → routes within a region automatically share the same type.
- Mixing can occur in edge cases (cross-region, special config) but is rare in practice.
- FCTD is the strictest — dedicated to one client by design.

> **⚠️ Don't assume hard rules**
> Older docs said 'a driver cannot book a route mixing different client types'. In practice this is mostly true because regions enforce single-type, but it's not an absolute code-level block for NEXT_DAYS / ON_DEMAND / SPECIALTY. Always verify the actual config when testing.

### 4.4 Quick Comparison

| Criterion | NEXT_DAYS | ON_DEMAND | SPECIALTY | FCTD |
| --- | --- | --- | --- | --- |
| Code | 0 | 1 | 2 | 3 |
| Pickup location | Jitsu Main WH | Client WH/Store | Client WH/Store | Client WH |
| Goes through Inbound Sort? | Always | Optional | Optional | Optional |
| Commingle with other clients? | Yes | Yes (rare) | Yes (rare) | Never (dedicated) |
| Speed | 8AM–8PM | Same-day / rapid | Special | Per contract |
| Pricing | Lowest | High | Highest | Negotiated |

### 4.5 Client Hierarchy & Brand Structure

Beyond service types, Jitsu supports 3 different ways a Client can be structured. A single business entity can have multiple sub-brands sharing infrastructure but with different visibility/billing rules.

> **📌 Real example — Hello Fresh (Client ID 49)**
> Hello Fresh has all 3 client types under one umbrella — making it the canonical example of how Jitsu's client hierarchy works in practice.

**1️⃣ Parent Client**
- Standalone main client with its own account.
- Has its own login credentials for the Client Web.
- Has its own unique `client_id`.
- Can submit shipments via both CSV upload and API.
- Independent billing — own `purchase_id`.

**2️⃣ Brand (under Parent Client)**
- Created FROM the Parent Client's account with a different brand name.
- Does NOT have its own login account for Client Web — shares with the Parent Client.
- Does NOT have its own `client_id` — uses Parent's ID.
- Distinguished only when submitting shipments — must include `shipment_code` in CSV or API payload to identify whether the shipment belongs to Parent or to a specific Brand.
- The `shipment_code` determines which brand name appears on the tracking page, label, reports, etc.
- Billing rolls up to the Parent Client.

> **⚠️ shipment_code is the key for Brand differentiation**
> When a CSV/API doesn't include `shipment_code`, the system treats all shipments as belonging to the Parent Client. To split shipments across brands, the `shipment_code` field MUST be present in every shipment record.

**3️⃣ Brand but has client_id (independent login)**
- Has its OWN distinct `client_id` (looks independent from Parent).
- CAN login to Client Web as a normal client — has its own account credentials.
- Still has `shipment_code` for differentiation when needed.
- BUT — `purchase_id` is linked to the Parent Client — meaning billing/invoicing still rolls up to the Parent.
- Use case: a larger sub-brand with its own ops team that needs independent access, while finance stays centralized at the Parent.

**Quick Comparison**

| Attribute | Parent Client | Brand | Brand but has client_id |
| --- | --- | --- | --- |
| Own client_id? | ✅ Yes | ❌ No (shares Parent's) | ✅ Yes (own ID) |
| Login Client Web? | ✅ Yes — own account | ❌ No — via Parent only | ✅ Yes — own account |
| How to differentiate when submitting? | By client account/token | By `shipment_code` in CSV/API | By client account + `shipment_code` |
| purchase_id (billing) | Independent (own) | Rolls up to Parent | Linked to Parent's purchase_id |
| How DB distinguishes Brand vs Brand-with-id? | — | No own client row — only `shipment_code` mapping | Has own client row, but `purchase_id` field links to Parent |

**Implications for QA**

- When testing display logic (tracking page, labels, reports) — always check the `shipment_code` field. The brand name shown to end users depends on it.
- When testing billing/reports — remember that `purchase_id` may be aggregated to the Parent even though shipments come from a Brand.
- When setting up test data — for Brand testing, you must include `shipment_code` in your CSV uploads or API payloads.
- When testing Client Web login — only Parent Clients and 'Brand with client_id' can login. Pure Brands cannot.
- If a ticket says 'shipment for Brand X', verify what type Brand X is in the DB before assuming the flow.

> **🔍 Production reference**
> Hello Fresh (Client ID 49) is the canonical example — it has Parent Client setup plus multiple Brands and at least one 'Brand with client_id'. When in doubt, check how Hello Fresh's brands are configured for reference.

## 5. Driver Types & How They Get Routes

Jitsu does not employ drivers directly. There are 4 distinct types — each with different rules for how they receive work.

### 5.1 The 4 Driver Types

| Driver Type | Full Name | Role |
| --- | --- | --- |
| **IC** | Independent Contractor | Individual drivers who book delivery opportunities through the Driver App. Restricted by zone. Last-mile only. |
| **DSP** | Delivery Service Provider | Contracted delivery companies covering negotiated areas. Drivers managed through their own DSP portal. Not bound by zones. |
| **3P** | Third-party gig platforms | Drivers from labor platforms (WorkWhile, InstaWork, Wonolo). Used as a last resort when IC + DSP coverage is insufficient (peak season, surges). |
| **Linehaul** | Linehaul Driver | Transports first/middle mile freight: from Client to Jitsu warehouses, or between Jitsu warehouses. Moves pallets and truckloads (not individual parcels). |

> **📌 'Courier' terminology**
> In Jitsu, 'courier' refers ONLY to DSP and 3P drivers. IC drivers are NEVER called couriers.

### 5.2 How Each Driver Type Gets Routes

There are 3 main ways a driver receives a route — but not all driver types can use all methods:

- **Method 1: Assign route/ticket directly to driver** — A dispatcher or admin assigns a route/ticket to a specific driver — no booking required.
- **Method 2: Book route from Direct Booking** — Driver sees a route with full details (exact pay, location) and books it.
- **Method 3: Book ticket from Booking Session** — Advance booking — driver books tickets in a zone ahead of time, only seeing pay range and parcel count.

> **🎯 Critical permissions rule**
> Only IC drivers can book from Booking Session and Direct Booking. DSP and 3P drivers are assigned routes directly — either from their own DSP board, or by a Jitsu dispatcher.

### 5.3 Permission Matrix

| Method | IC | DSP | 3P | Linehaul |
| --- | --- | --- | --- | --- |
| Booking Session (advance) | ✓ Yes | ✗ No | ✗ No | — |
| Direct Booking | ✓ Yes | ✗ No | ✗ No | — |
| Direct Assign by Jitsu Dispatcher | ✓ Yes | ✓ Yes | ✓ Yes | — |
| Assign from DSP Board | ✗ No | ✓ Yes | ✓ Yes | — |

> **ℹ️ Linehaul Driver — quick note**
> Linehaul drivers operate on a completely separate workflow (NetSuite-driven auto-assignment, BOL-based pickup/dropoff). They do not participate in any of the booking methods above. Detailed Linehaul testing has its own dedicated documentation.

### 6.4 Mobile Apps

Jitsu has 4 mobile apps for field operations. Mobile apps do not have web URLs — they release by version.

| Mobile App | Purpose |
| --- | --- |
| **Driver App (Jitsu Drive)** | Used by IC, DSP, 3P drivers: registration, booking, pickup, delivery, POD, payment screens. |
| **Outbound App** | Used by Outbound staff: yard management, ID scan, route check-out to drivers. |

**Mobile App Release & Testing Workflow** — Mobile apps are released by version; testing is environment-staged, not URL-based:

1. On Staging — download builds from Firebase to test.
2. All tickets in a version must PASS on Staging.
3. Once all pass, dev pushes the build to TestFlight (the Production environment for mobile).
4. Test the build on TestFlight (Production).
5. If all tickets pass on Production → version is READY to release.

> **📌 NOTE — Mobile testing scope**
> Mobile apps are tested ONLY on Staging and Production (TestFlight). There is no Beta build for mobile — the Beta environment is for web apps only.

## 8. QA Process & Workflow

Follow this process for every assigned ticket. Consistency is what keeps the team efficient.

> **🎯 GOLDEN RULE for Maintain Tickets**
> When you receive a maintain ticket (not a new feature), first check the current flow on Production and look at REAL DATA. Understand how end users actually behave today — then build test cases from the end-user viewpoint to extract maximum coverage from the ticket.

**Step 1 — Understand the Ticket**
- Read the user story and acceptance criteria carefully.
- Identify which part of the delivery flow it touches.
- Identify which driver type / client type / warehouse is involved.

If it's a Maintain ticket:
1. Check the current flow on Production first.
2. Look at REAL data — what does end-user behavior actually look like?
3. Build the test case from an end-user point of view.
4. Goal: extract maximum coverage by exploring the ticket from real usage patterns.

**Step 2 — Write Test Cases**
- Write test cases using the Confluence-attached format (see Section 9).
- Upload them to QMetry under the Test Case tab.
- Test cases are organized by Application.

Folder structure in QMetry:
1. Navigate to the correct Application.
2. Find the relevant function.
3. Create a folder named with the ticket ID (e.g. ENG-6799).
4. Import the prepared test case file, or create test cases directly.

**Step 3 — Test Cycle**

For tickets that have test cases, testing must be executed through Test Cycles in each environment. Sprints or app versions are already created in the Test Cycle tab.

Web App tickets (tested by Sprint):
1. Navigate to the correct Sprint.
2. Create a new folder named with the ticket ID.
3. Inside the folder, create test cycles for each environment.

Naming convention:
```
Folder name: Ticket_ID
Cycle name:  [Environment]_[Ticket_ID]_Ticket name
```
Example:
```
Folder: ENG-6799
Cycle (Staging): [Staging]_[BE] Update API to support selection of reason codes for unbooked tickets
```

**Step 4 — Verification Comments on Jira**

> **⚠️ IMPORTANT**
> When verifying on Staging, make sure the configuration in Staging matches Production. On Beta, the config should also match Prod. During testing, watch the Datadog logs to make sure no errors appear after a new deploy.

Every Jira ticket needs a clear verification comment containing:
- Tested environment(s).
- Test result: PASSED or FAILED.
- Clear explanation of the result.
- Evidence attached (main flow is sufficient).

Example comment: `Tested on Staging — PASSED`. If a ticket cannot be tested on Beta or Production, clearly state the reason in the comment.

**Step 5 — Tickets Without Test Cases**

For tickets that do not require formal test cases (verified via a checklist): clearly list all verified scenarios in the Jira comment.

**Step 6 — Documentation**

For tickets that introduce new configuration or logic without existing documentation: QA is encouraged to create the documentation. AI tools may be used to assist with writing.

**Step 7 — Reporting**

*Daily QA Report:*
- Update the QA tracking file (Excel) daily.
- Post updates in the QA channel so the lead can track progress.
- If unable to update the file daily, proactively report to the QA lead.

*Mobile Testing — Version Completion Report:*
- Actively track the progress of the app version.
- Once all assigned tickets pass on Staging → notify the qa-mobile channel and request push to TestFlight (Production).
- Once all pass on Production → version is READY to release.

### Step 8 — Bug Reporting

When testing reveals an issue, log a bug ticket on Jira. Jira has pre-defined fields — fill them according to the template below.

**Required Fields on Jira**

| Field | Required? | Content to fill |
| --- | --- | --- |
| **Title (Summary)** | ✅ | Format: `[Component/Feature] - [Actual issue]` |
| **Environment** | ✅ | Staging / Beta / Production |
| **Description** | ✅ | 1–2 sentences describing the phenomenon concisely |
| **Preconditions** | ✅ | Config flags + test data + role needed |
| **Steps to Reproduce** | ✅ | Specific steps; a dev following them must reproduce 100% |
| **Actual Result** | ✅ | The WRONG behavior — describe objectively, don't speculate on cause |
| **Expected Result** | ✅ | Reference the AC or business logic |
| **Evidence** | ✅ | Screenshot/Video/Datadog log/Test data IDs. No evidence = ticket bounced |
| **Issue Type** | ✅ | Must select Bug |
| **Priority** | ✅ | By business impact (Critical/High/Medium/Low) |
| **Assignee (Dev)** | ⚠️ Optional | Dev ownership component (if known) |
| **QA Review** | ⚠️ Optional with rule | See the QA Review rule below |

> **🎯 IMPORTANT rule about the QA Review field**
> **Case 1 — Logging a standalone bug:** QA Review = Optional, leave blank for PO/Lead to assign later.
> **Case 2 — Logging a bug DURING testing of a ticket you are assigned:** QA Review = the QA who logged the bug (REQUIRED). Reason: you're testing ticket A → find a bug → log bug B. When dev fixes bug B, you verify it because you understand the context of ticket A.

**📱 Mobile Bug — Additional required fields**

When logging a bug for mobile apps (Driver / Outbound / Warehouse / Linehaul), also fill in:

```
App & Version:
- App: [Driver App / Outbound App / Warehouse App / Linehaul App]
- Current version: [e.g. 2.8.2, the version the bug was found on]
- Build source: [Firebase Staging build / TestFlight Prod / Play Store]
Fix Version: [Version where this bug will be fixed, e.g. 2.9.0]
```

Why Mobile needs these fields:
- **App + Current Version:** A bug may only occur on a specific version (e.g. MOB-2643 only on v2.8.0). Dev needs the version to reproduce and identify regression.
- **Build source:** Distinguishes Staging Firebase vs TestFlight (Prod) — behavior may differ.
- **Fix Version (critical):** The target version that will release the fix. Tracks the release roadmap; mobile QA knows which version to re-test; mobile has its own release cycle by version.

**📊 Comparison: Bug Mobile vs Bug Web**

| Aspect | Bug Mobile | Bug Web |
| --- | --- | --- |
| App identifier | App name + Current Version + Build source | Browser + Version (note in Evidence) |
| Environment | Staging Firebase / TestFlight Prod | Staging / Beta / Prod URL |
| Fix Version | ✅ Required (track release cycle) | ❌ Not needed (web deploys continuously) |
| Preferred Evidence | Screen recording + Datadog log | Console errors + Network logs (DevTools F12) |
| User identity | (already in test data IDs) | Username + Scopes (note in Preconditions) |

**✅ Checklist before submitting a Bug**

*📝 Content:*
- Title clear format: `[Component] - [Issue]`
- Environment specified (Staging/Beta/Prod)
- Steps to Reproduce can be followed to reproduce 100%
- Actual ≠ Expected — written separately, not merged
- Evidence attached (screenshot/video at minimum)

*📱 If Mobile:*
- App + Current version + Build source
- Fix Version (version that will release the fix)

**🎓 Common Mistakes — Avoid these**

1. Vague title: "Bug in Driver App" → ✅ "Driver App crashes when tapping POD on Android"
2. Description = Steps to Reproduce → These are 2 DIFFERENT fields. Description = a 1–2 sentence summary.
3. No Expected Result → Dev doesn't know where the bug is. Always reference the AC.
4. Speculating on cause: "Bug due to API error" → Describe the phenomenon, let dev investigate root cause.
5. No evidence attached → Ticket bounced immediately.
6. Skipping Preconditions → Dev wastes 30 minutes setting up.
7. Forgetting to link the parent ticket + forgetting to set QA Review = yourself when the bug is from a ticket you're testing.

> **📱 Mobile-specific tip**

> **🖥 Web-specific tip**
> Web deploys continuously (no version cycle like mobile) → no Fix Version needed. Instead, Console + Network logs are the strongest evidence → paste into the Evidence field.

## 11. End-to-End Testing Guide

This section walks you through a complete E2E flow. As a new QA, follow it in order at least once on Staging — that will give you a working mental model of how Jitsu actually moves a parcel from a client all the way to a recipient.

> **🎯 GOAL**
> By the end of this section, you should be able to: create a Client → submit a shipment → route it → assign a driver who books and delivers it → simulate payment.

**E2E Flow Overview**
1. Create a Client (Dashboard)
2. Submit a Shipment (Client web — batch or single API)

### 11.0 Database & Tools Access via StrongDM

> **🔐 IMPORTANT — Required before E2E testing**
> StrongDM is how QA accesses Jitsu's protected resources: databases (PostgreSQL, MongoDB, Cassandra), Jenkins, internal web apps, Consul, Nexus repositories, and SSH to servers. Set this up BEFORE starting the E2E flow.

**What is StrongDM?** A proxy that manages and audits access to protected resources such as databases, servers, clusters, and web apps. It replaces traditional VPN setup. You won't need to remember resource addresses or credentials — StrongDM handles it all. The local client tunnels requests from your workstation to the gateway, so you only need to connect to `localhost`.

**Installation — StrongDM Desktop Client**
1. Login to https://app.strongdm.com/ and go to the Download & Install page.
2. Click 'Download StrongDM for macOS' (or your OS) to get the desktop app and CLI.
3. Install the application.
4. Open the SDM client — login with your Jitsu credentials.

The 'sdm' CLI is optional, for those who prefer a command-line workflow (required for SSH/SCP use cases).

**Basic Usage — Connect to a Resource**
1. Open the SDM desktop client.
2. Find the resource you need (e.g. 'staging-postgresql', 'staging-mongodb').
3. Click the resource → wait until status shows ♥ Connected.
4. Now you can access the resource via `localhost` + the corresponding port.

> **💡 Resource access requests**
> If you don't see a resource you need, submit a request in SDM. DevOps/IT must approve. Allow 1–2 business days.

**Connect to Internal Websites (`*.gojitsu.sdm.network`)** — Many internal tools (Jenkins, Consul UI, Nexus) are accessible only through the StrongDM proxy. On macOS: System Preferences → Network → select your connection → Advanced → Proxies tab → check 'Automatic Proxy Configuration' → add URL `https://app.strongdm.com/proxy.pac` → OK → Apply.

**Connect to PostgreSQL** (DBeaver, pgAdmin):
- **Host:** localhost
- **Port:** the port shown next to the resource in the SDM client (varies per resource)
- **Database:** can be left empty (or specify the DB name like `axlehire_dataorch`)
- **Authentication:** can be omitted (SDM handles it automatically)

> **💡 Troubleshooting — DBeaver shows no tables?**
> Enable 'Show all databases' in connection settings, OR specify the correct database name in the Database field.

**Connect to MongoDB** (Studio 3T, MongoDB Compass): Host=localhost, Port from SDM, no auth required. Common collections you'll query: `sort_center` (sortation center configs), `driver_registration_record` (driver BG check status), `measurement_obligations` (dimension records).

**Access Jenkins:** ensure browser proxy is configured → connect to Jenkins resource in SDM → open `https://jenkins.gojitsu.sdm.network/`.

**Common Troubleshooting:**
1. *Resource shows 'Disconnected':* ensure the SDM client is running and you're logged in; restart it; check internet; request DevOps to verify permissions.
2. *PostgreSQL DBeaver shows no tables:* enable 'Show all databases' or specify the correct database name.
3. *Internal website won't load:* verify the system proxy (`https://app.strongdm.com/proxy.pac`); make sure the resource is ♥ Connected; clear browser cache.
4. *NPM/Maven downloads fail:* verify the Nexus resource is connected; check the proxy port matches SDM; restart the package manager.

### 11.1 Setup: Create a Client

> **📌 Precondition:** You must have an Admin account that can login to the Dashboard (see Section 11.0.5).

1. Open the Dashboard (Staging): https://dashboard.staging.gojitsu.com/
2. Select **Client** in the left menu.
3. Click **Add** — the 'ADD NEW CLIENT' pop-up appears.
4. Enter all valid information; choose one of the 4 client service types (NEXT_DAYS / ON_DEMAND / SPECIALTY / FCTD).
5. After creation, add the client service directly into the `client_service` table in PostgreSQL.

> **📌 NOTE — Region/Type matching**
> For each region you want the client to serve, you must add a corresponding entry into the `client_service` table. Since ONE region = ONE client service type, the region you select must match the client's type.

**Decide the Client Setup Type First** (see Section 4.5 for theory):
- **Parent Client (Type 1):** Create the client normally on Dashboard with own credentials. Add `client_service` entries for each region it serves. No special fields needed. Useful for testing standard shipment submission, login, billing flows.
- **Brand (Type 2 — no own client_id):** Create the Parent Client first. Add the Brand under the Parent (usually at the brand/shipment level, not a new client row). ALWAYS include `shipment_code` in CSV/API when submitting test shipments. Verify display in tracking page, label, reports. Cannot login as the Brand.
- **Brand with own client_id (Type 3):** Create the Brand as a separate client row (gets its own `client_id`). Configure login credentials (can login independently). In DB, set `purchase_id` to link back to the Parent. Add `client_service` entries. Verify billing/reports aggregate at the Parent level.

> **💡 Best practice for QA test setup**
> For most tickets, a simple Parent Client (Type 1) is enough. Only set up Brands (Type 2 or 3) when the ticket specifically involves brand differentiation, multi-brand display logic, or billing aggregation testing. To see how all 3 types coexist, look up Hello Fresh (Client ID 49) in Production — don't modify Hello Fresh data, just use it as reference.

### 11.9 Scan All Shipments of a Solution Automatically (Postman)

This automates inbound scanning when you have a solution ID. Example: solution ID 56433.

1. Create a new folder, e.g. 'scan all shipments in solution'.
2. Add a GET request inside it: `{{routing_app_url}}/routing/solutions/{{solutionId}}/assignments`

   Tests tab script:
   ```javascript
   var jsonData = pm.response.json();
   var listAssignments = jsonData.assignments.map(item => item.id);
   pm.environment.set("assignmentIds", JSON.stringify(listAssignments));
   pm.environment.set("shipmentIds", "[]");
   ```
3. Add a request named 'getShipment': `{{dataorch_url}}/assignments/{{assignmentId}}/detail`

   Pre-request Script:
   ```javascript
   var list = JSON.parse(pm.environment.get("assignmentIds"));
   pm.environment.set("assignmentId", list.shift());
   pm.environment.set("assignmentIds", JSON.stringify(list));
   ```
   Tests Script (chains to next request if more assignments remain):
   ```javascript
   var jsonData = pm.response.json();
   let shipments = jsonData.shipments.map(item => item.id);
   let shipmentIds = JSON.parse(pm.environment.get("shipmentIds"));
   shipmentIds.push(...shipments);
   pm.environment.set("shipmentIds", JSON.stringify(shipmentIds));
   let assignmentIds = JSON.parse(pm.environment.get("assignmentIds"));
   if (assignmentIds.length > 0) postman.setNextRequest("getShipment");
   ```
4. Add a request named 'scanShipment': `{{inbound_api_url}}/info/{{shipmentID}}`

   Pre-request:
   ```javascript
   var shipmentList = JSON.parse(pm.environment.get("shipmentIds"));
   pm.environment.set("shipmentID", shipmentList.shift());
   pm.environment.set("shipmentIds", JSON.stringify(shipmentList));
   ```
   Tests:
   ```javascript
   pm.test("Status code is 200", function () { pm.response.to.have.status(200); });
   let shipmentList = JSON.parse(pm.environment.get("shipmentIds"));
   if (shipmentList.length > 0) postman.setNextRequest("scanShipment");
   ```
5. Right-click the folder → Run Collection.

### 11.11 Test Labels

Label version is decided by config keys in MongoDB. The selection rule:

| Has parcel? | parcel_label_version + label_version | parcel_label_version only | label_version only | No config |
| --- | --- | --- | --- | --- |
| Yes (has parcel) | parcel_label_version | parcel_label_version | label_version | postgres config |
| No (no parcel) | label_version | postgres config | label_version | postgres config |

- **Download Label — Public API (Client app, single shipment):** PDF — get version from Postgres first; if shipment has parcels, use mongo `parcel_label_version`. ZPL — get version from mongo `zpl-label-version`; if missing, return DEFAULT (hardcoded).
- **Worker (PDF, multiple shipments):** `LabelClientShipmentWorker`: mongo `parcel_label_version` (high priority) → `label_version` (medium). `LabelJobWorker`: `presets.version` → fallback to Postgres.
- **Data Orchestra (PDF) — Dispatch App:** Shipment has parcels → `parcel_label_version`. No parcels → `zpl-label-version` → fallback to Postgres.
